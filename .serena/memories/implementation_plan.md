# Pretix Bevy Integration Implementation Plan

This memory outlines the technical architecture and detailed implementation steps for connecting Pretix events to Bevy using an asynchronous, event-driven approach.

## 1. Architectural Strategy
Use an **event-driven asynchronous approach**.

- **Trigger:** Hook into Pretix signals (`order_paid` and `checkin_created`).
- **Processing:** Signal handlers enqueue Celery background tasks instead of blocking the Pretix request cycle.
- **Execution:** Background tasks execute Bevy API calls using credentials stored in Pretix settings.
- **Reliability:** Use idempotency checks/logging around each sync attempt so duplicate signals or task retries do not silently create duplicate Bevy attendees.

## 2. Configuration Settings (django-hierarkey)
Pretix uses `django-hierarkey`, allowing settings to cascade from Global â†’ Organizer â†’ Event.

### Global Level
- **`bevy_api_base_url`**: Bevy API base URL, defaulting to `https://gdg.community.dev/api`.
- **Implementation:** In `bevy/signals.py`, connect to `pretix.base.signals.register_global_settings` and return an `OrderedDict` containing a `forms.URLField`.

### Organizer Level
- **`bevy_chapter_id`**: Default Bevy Chapter ID for all events under this organizer.
- **`bevy_cookie_json`**: Preferred auth input. A JSON list of cookie objects copied from a Playwright/Chrome storage state or generated by the helper script/service. Store the structured cookies instead of only a raw `Cookie` header so expiry, domain, and token extraction can be validated.
- **`bevy_cookie`**: Fallback/manual override raw cookie string for emergency use.
- **`bevy_csrf_token`**: Optional manual override. If empty, derive it from the `csrftoken` cookie in `bevy_cookie_json` or `bevy_cookie`.
- **`bevy_auth_last_validated_at`**: Read-only/diagnostic setting updated after a successful auth validation.
- **Implementation:**
  - Create `bevy/forms.py` with an `OrganizerSettingsForm` containing chapter ID and auth fields.
  - Prefer a textarea for `bevy_cookie_json`; validate that it is JSON, contains at least `csrftoken`, and ideally contains the authenticated session cookie.
  - Create `bevy/views.py` with an `OrganizerSettingsView` inheriting from `OrganizerSettingsFormView`.
  - Connect it via `nav_organizer_settings` in `bevy/signals.py`.
  - Add URL routing in `bevy/urls.py`.

### Event Level
- **`bevy_event_id`**: Specific Bevy event ID.
- **Implementation:**
  - Create `EventSettingsForm` in `bevy/forms.py`.
  - Create `EventSettingsView` inheriting from `EventSettingsFormView` in `bevy/views.py`.
  - Connect it via `nav_event_settings` in `bevy/signals.py`.
  - Add URL routing in `bevy/urls.py`.

## 3. Cookie / Session Generation Strategy
Manual raw cookie entry is fragile. The improved minimal strategy is:

1. **Generate cookies outside Pretix** using the existing Bevy auth flow:
   - A human logs into `https://gdg.community.dev` in Chrome.
   - The existing Playwright/CDP auth flow captures browser `storage_state`.
   - A small helper extracts the `cookies` array and provides either:
     - structured JSON for `bevy_cookie_json`, or
     - a derived raw `Cookie` header and CSRF token for fallback fields.
2. **Store structured cookie JSON in Pretix organizer settings** rather than relying only on a pasted raw header.
3. **Build request auth at runtime**:
   - Parse `bevy_cookie_json`.
   - Filter relevant cookies for `gdg.community.dev`.
   - Derive `csrf_token` from the `csrftoken` cookie unless `bevy_csrf_token` is explicitly set.
   - Build the `Cookie` header as `name=value; name=value`.
4. **Validate before saving or via a button/task**:
   - Call a lightweight authenticated endpoint such as `GET /chapter/{chapter_id}/` or event detail.
   - Save/update `bevy_auth_last_validated_at` only on success.
   - Show clear admin feedback if auth is invalid or expired.
5. **Handle expiry intentionally**:
   - Inspect cookie `expires` values when available and warn if any required cookie is expired/near expiry.
   - On HTTP `401` or `403`, do not blindly retry many times. Log a clear warning and mark auth invalid/expired so the organizer refreshes the cookie JSON.

This keeps cookie generation simple and human-controlled while making the plugin safer than raw copy-paste headers.

## 4. Unified Ingestion Task (Celery)
Define a single background task in `bevy/tasks.py`.

- Use `pretix.celery_app.app`.
- Pass only JSON-serializable task arguments: event primary key, order position primary key, and action.
- Fetch `Event` and `OrderPosition` inside the task.
- Extract settings via `event.settings.get(...)`.
- Build Bevy headers with:
  - `accept: application/json; version=bevy.1.0`
  - `content-type: application/json`
  - `origin: https://gdg.community.dev`
  - `x-requested-with: XMLHttpRequest`
  - `cookie: <derived cookie string>`
  - `x-csrftoken: <derived csrf token>`
- Use `requests.Session` with a timeout for API calls.
- For registration, `POST {base_url}/attendee/?event={bevy_event_id}&chapter={chapter_id}` with payload:
  - `event`
  - `attendees`: first name, last name, email, `is_checked_in: false`, `send_event_email: true`
- For check-in, Bevy requires Bevy attendee IDs. Therefore, before `PUT /attendee/checkin/`, search by email with `GET /attendee_search/?event=...&chapter=...&search=<email>` and select the exact matching attendee. If none is found, either register first or log and retry after registration completes.
- Error handling:
  - Missing config: log and abort gracefully without retry.
  - `401`/`403`: log auth expired/invalid and abort or do one short retry only if validation suggests transient auth propagation.
  - `429` and `5xx`: retry with Celery exponential backoff/jitter.
  - Other `4xx`: log response body and abort without retry.

## 5. Signal Receivers
In `bevy/signals.py`, listen to Pretix execution signals.

### `order_paid`
- Receiver dispatch UID: `bevy_order_paid`.
- For each order position with an email or fallback order email, enqueue `sync_attendee_to_bevy.apply_async(args=(sender.pk, position.pk, "register"))`.

### `checkin_created`
- Receiver dispatch UID: `bevy_checkin_created`.
- Enqueue `sync_attendee_to_bevy.apply_async(args=(sender.pk, checkin.position.pk, "checkin"))`.

## 6. Logging and Validation Before Fixing Issues
For any sync failure, first validate likely causes with logs before changing logic. The likely causes are:

1. Missing or wrong `bevy_event_id` / `bevy_chapter_id`.
2. Expired or malformed Bevy cookies / CSRF token.
3. Bevy attendee already exists or duplicate registration behavior.
4. Check-in task cannot map Pretix position to Bevy attendee ID.
5. Bevy API schema/permission change.
6. Network timeout/rate limit/server error.

Log minimally but clearly:
- Pretix event ID, order position ID, action.
- Bevy event/chapter ID.
- Response status code and truncated response body.
- Never log the full cookie string or CSRF token.

## 7. Implementation Order
1. Settings forms/views/URLs/navigation.
2. Cookie helper functions: parse structured cookies, derive CSRF, build headers, validate expiry.
3. Bevy API client helper with timeout and safe error handling.
4. Celery task for register/check-in.
5. Signal receivers.
6. Manual validation using one paid order and one check-in.
